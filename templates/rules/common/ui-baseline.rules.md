# UI Baseline RL（最低保証・常時適用）

> ROLE: これは人間が知覚・操作しやすいUIを最低限保証するための不変ルール。
> SCOPE: 全画面・全コンポーネントに常時適用。目的非依存。
> SEVERITY: `MUST` 違反は欠陥（マージ不可）。`SHOULD` 違反は要正当化。
> 特化目的による緩和は `ui-specialization.context.md` の安全境界に従う。それ以外で本ファイルを緩和しない。
> 各ルールの末尾 `∵` は根拠（神経科学 / UX法則）。CCが理由を理解し判断ブレを減らすために付す。
> ORIGIN: external（利用者提供の UIUX 研究、dialog-harness v5.23.0 で取り込み）。DH における適用経路（DESIGN.md 視覚トークン層との二層関係 / 5層検出スタックの第2・5層 oracle 化 / independent-reviewer のレビュー観点）は DH 側 `.claude/skills/layer0-spec-architect/references/design-system-spec.md` の「UI 相互作用層」節を参照。

## 0. 語彙（混同禁止）

設計指示・レビューでは以下を厳密に区別する。「アフォーダンスを追加する」は誤用。

| 用語 | 定義 | スクリーンUIでの扱い |
|------|------|------|
| affordance | 対象が許す行為そのもの（知覚されるか否かに依存しない） | 物理操作（タッチ/ドラッグ）にのみ真に存在 |
| perceived affordance | ユーザーが「可能」と感じる行為 | 設計者が制御する対象の本体 |
| signifier | 行為可能性を告げる知覚的手がかり | 立体感・影・下線・カーソル変化・ラベル |
| convention | 学習済みの記号解釈（慣習） | フラットUIのクリック可能性はこれに依存 |
| false affordance | 操作可能に見えて不可能 / 逆 | 不具合の主要分類。検出対象 |

∵ Norman は2013改訂版で affordance と signifier を分離。「画面にアフォーダンスを加えた」は実際には signifier か convention の操作であり、affordance ではない。

---

## 1. signifier（操作可能性の可視化）

- **B-01 `MUST`** 操作可能要素は、静止状態で操作可能だと分かる signifier を1つ以上持つ（色・形・配置・ラベルのいずれか）。hover/focus でしか現れない signifier は単独では不可。 ∵ 背側経路は無意識かつ自動でアフォーダンスを知覚する。手がかりが無いと行動が起動しない。
- **B-02 `MUST`** 操作不可能な要素は、操作可能に見えてはならない（false affordance 禁止）。装飾の枠線・影をボタン以外に付けない。 ∵ false affordance は誤クリックと不信の主因。
- **B-03 `SHOULD`** 同一の signifier は同一の意味に対応させる（色＝意味の1対1）。同じ青を「リンク」と「無効状態」に併用しない。
- **B-04 `MUST`** インタラクティブ要素には状態 signifier を与える（hover / focus / active / disabled / loading）。最低でも focus と disabled。 ∵ 状態の不可視化は予測誤差を生む（§5）。

## 2. convention（慣習遵守）

- **B-05 `MUST`** 標準パターンを再発明しない（ナビ位置・アイコン意味・フォーム挙動・戻る動作）。逸脱には明示的理由を残す。 ∵ Jakob's Law：ユーザーは他サービスで大半の時間を過ごし、期待をそこで形成する。
- **B-06 `SHOULD`** プラットフォーム規約（iOS HIG / Material / Web標準）に整合させる。混在させない。
- **B-07 `MUST`** アイコン単独で意味を伝えない。曖昧なアイコンにはラベルかツールチップを併置。 ∵ アイコン解釈は convention 依存で、共有されない慣習は通じない。

## 3. cognitive load（認知負荷）

- **B-08 `MUST`** 1画面・1セクションの同時選択肢を絞る。主要アクションは1〜2に限定し、視覚階層で序列化する。 ∵ Hick's Law：選択肢数に対し決定時間が対数的に増える。
- **B-09 `SHOULD`** 関連項目をグルーピングしチャンク化する。並列の独立項目を概ね5〜9以内に保つ。超える場合はグループ化・段階開示。 ∵ Miller's Law：作業記憶は約7±2項目。
- **B-10 `MUST`** 「認識 > 想起」。ユーザーに記憶させず、選択肢・履歴・既定値を提示する。 ∵ 想起は作業記憶を消費し負荷が高い。
- **B-11 `SHOULD`** 不要な要素を排除（視覚ノイズ・重複導線・過剰装飾）。情報は段階開示（progressive disclosure）。

## 4. interaction mechanics（操作機構）

- **B-12 `MUST`** タップ/クリック対象は十分大きく、間隔を空ける（タッチ目安 44×44px 以上）。 ∵ Fitts's Law：到達時間はターゲットの大きさと距離で決まる。
- **B-13 `SHOULD`** 頻用・重要アクションを近く・大きく、破壊的アクションを遠く・小さく配置する。 ∵ Fitts's Law の意図的運用＝誤操作コスト設計。
- **B-14 `MUST`** クリック判定領域を可視領域と一致させる（ラベルとヒット域のズレ禁止）。 ∵ false affordance の一種。

## 5. feedback / 予測誤差（即時応答）

- **B-15 `MUST`** 全操作に即時の知覚的応答を返す（押下表現・状態変化・トースト等）。応答なしの操作を作らない。 ∵ 能動的推論：UIはユーザーの予測誤差を減らす装置。無応答は誤差を放置する。
- **B-16 `MUST`** 100ms超かかる処理は進行表示（spinner / skeleton / progress）。可能なら楽観的更新（optimistic UI）。 ∵ 予測誤差の先回り。Doherty Threshold（応答<400msで作業が滑らかになる）。
- **B-17 `MUST`** エラーは「何が起きたか・どう直すか」を、発生箇所の近くで人語で示す。コード/スタックを生で出さない。
- **B-18 `SHOULD`** 不可逆操作は取り消し（undo）か確認を用意する。

## 6. visual structure（視覚構造）

- **B-19 `MUST`** 視覚階層を確立する（サイズ・ウェイト・色・余白で主従を表す）。全要素を等価に並べない。 ∵ 腹側経路の物体認識＋Gestalt。
- **B-20 `SHOULD`** Gestalt原則を意図的に使う：近接（関連は近く）・類似（同種は同形）・共通領域（境界で囲う）。配置と意味を一致させる。
- **B-21 `MUST`** 本文コントラストを確保（WCAG AA：通常文字 4.5:1、大文字 3:1 以上）。色のみで情報を区別しない。 ∵ 知覚の最低線。色覚多様性・低視力に必須。

## 7. accessibility floor（最低線・特化目的でも侵食不可）

- **B-22 `MUST`** 全機能をキーボードのみで操作可能にする。focus 可視・論理的タブ順。
- **B-23 `MUST`** 画像/アイコン/コントロールに代替テキスト・aria 名を付与。意味を持つ要素を装飾扱いしない。
- **B-24 `MUST`** アニメーションは `prefers-reduced-motion` を尊重する。点滅（3回/秒超）禁止。
- **B-25 `SHOULD`** フォーム入力に明示ラベル（placeholder をラベル代替にしない）・適切な input type / inputmode。

## レビューチェックリスト（PR / 保守時）

各項目を画面単位で yes/no 判定。no が出たら該当 B-ID を起票。

1. 操作可能要素はすべて静止状態で操作可能と分かるか（B-01 / B-14）
2. 操作可能に見えて不可能な要素はないか（B-02）
3. 状態（focus/disabled/loading）が可視か（B-04）
4. 標準パターンから逸脱していないか、逸脱に理由はあるか（B-05〜B-07）
5. 主要アクションは1〜2に絞られ序列化されているか（B-08）
6. 並列項目は概ね9以内、超過分はグループ化/段階開示されているか（B-09）
7. タップ域は十分か・破壊的操作は隔離されているか（B-12 / B-13）
8. 全操作に即時応答、遅延処理に進行表示があるか（B-15 / B-16）
9. エラーは近接・人語・対処提示か（B-17）
10. コントラストAA・色のみ依存なしか（B-21）
11. キーボード操作・focus可視・代替テキストは揃うか（B-22〜B-24）
