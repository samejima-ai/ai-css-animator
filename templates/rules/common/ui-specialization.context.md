# UI Specialization Context（目的特化・選択適用）

> ROLE: 特定の目的にUIを最適化するための可変context。
> SCOPE: 画面 / フロー / プロジェクト単位で目的を1つ選ぶ（または重み付け）。`ui-baseline.rules.md` を前提に上乗せする。
> 関係: Baseline と衝突しうる。各目的は「強化する／緩めてよい／緩めてはならない（安全境界）」を明示する。安全境界は Baseline でも侵食不可能な最低線。
> 用語・根拠表記は Baseline に準拠。
> ORIGIN: external（利用者提供の UIUX 研究、dialog-harness v5.23.0 で取り込み）。S-xx の選択は DH 側 L0 §3.6 DESIGN.md 対話 / UX 3問 / NFR スコアリングと連動する。利用者プロジェクトで S-xx を追加・上書きする場合は本ファイルを `.dh/rules/common/ui-specialization.context.md` に置いて override する（DH 更新で上書きされない）。

---

## 選択プロトコル

実装/設計の前に、対象画面の主目的を確定させてから本ファイルを適用する。

1. その画面の**主要ユーザー**は誰か（初見 / 熟練 / 混在）
2. **最重要の成功指標**は何か（速度 / 正確性 / 学習容易性 / 情報把握 / 継続 / 包摂）
3. その指標に対応する目的（S-xx）を1つ主・最大1つ副として選ぶ
4. 選んだ目的の「緩めてよいBaseline」だけを緩和し、理由を残す
5. 「緩めてはならない安全境界」は目的に関わらず保持する

複数目的が衝突したら §衝突解決 に従う。目的が未確定なら特化せず Baseline のみで実装する。

## S-01 高速反復入力（業務・熟練ユーザー）

- 想定: 同一ユーザーが日常的に大量入力する社内/業務系（例: タスク登録・出面入力・数量拾い）。
- 狙い: 操作回数とキーストロークの最小化。学習コストは許容。
- 強化: キーボード完結（tab/enter/ショートカット）、既定値・前回値の自動補完、連続入力モード、楽観的更新でテンポ維持。
- 緩めてよい: B-08/B-09（選択肢の絞り込み）— 熟練前提で密度を上げてよい。B-11（段階開示）— 全項目を一画面に出してよい。
- 緩めてはならない: B-15/B-16（即時feedback）、B-04（状態可視）、B-22（キーボード操作）— むしろ強化対象。
- 観測指標: 1件あたり入力時間、キーストローク数、誤入力率。

## S-02 エラー最小化・不可逆操作の保護

- 想定: 金額・契約・削除・発注など、誤りのコストが高い操作。
- 狙い: 誤操作の発生確率と影響の最小化。速度より正確性。
- 強化: 制約（入力フォーマット強制・選択式化）、確認・プレビュー、undo、危険操作の物理的隔離、入力後の即時検証と差分表示。
- 緩めてよい: 速度（多少の手数増は許容）、B-08を逆方向に（あえて一手間を足す）。
- 緩めてはならない: B-17（エラーの近接・人語・対処）、B-13（破壊的操作の隔離）、B-18（取り消し/確認）。
- 観測指標: 誤操作率、取り消し発生率、誤操作の事後コスト。

## S-03 初見学習容易性（オンボーディング・新規獲得）

- 想定: 初めて触るユーザーの離脱を防ぐ画面。
- 狙い: ゼロ知識で次の一手が分かる。
- 強化: 強い signifier とラベル併記、段階開示、空状態(empty state)で次アクション提示、デフォルトで安全な道、文脈ヘルプ。
- 緩めてよい: 情報密度（あえて少なく）、上級機能の即時露出（隠してよい）。
- 緩めてはならない: B-05〜B-07（慣習遵守）— 初見ほど慣習に依存する。B-01（signifier）。
- 観測指標: 初回完了率、初回離脱ポイント、ヘルプ参照率。

## S-04 高密度情報提示（ダッシュボード・専門家ツール）

- 想定: 大量データを一望し比較・判断する専門家画面。
- 狙い: 単位面積あたりの情報量と比較可能性の最大化。
- 強化: テーブル/小型multiple、整列とグリッド、データインク比の最適化、固定ヘッダ・フィルタ・ソート、密なタイポグラフィで走査性確保。
- 緩めてよい: B-09（Miller則の項目数）— 一覧性のため超過してよい。余白（詰めてよい）。B-11（段階開示）。
- 緩めてはならない: B-19/B-20（視覚階層・Gestalt）— 密度が上がるほど構造化が必須。B-21（コントラスト）。B-12は走査主体なら緩和可だが操作対象は維持。
- 観測指標: 目的データへの到達時間、比較タスク完了時間、誤読率。

## S-05 行動変容・継続（anticipatory / persuasive）

- 想定: 習慣化・完遂・再訪を促す画面（健康・学習・SaaS継続）。
- 狙い: 望ましい行動の継続。ただしユーザーのエージェンシー（制御・自律）を保つ。
- 強化: goal-gradient（進捗の可視化で完遂を促す）、peak-end（完了体験の山と締めを設計）、適切なリマインド、達成feedback。
- 緩めてよい: B-11（完遂導線を意図的に目立たせる）。
- 緩めてはならない: ユーザーの制御権（いつでも中断・解除・拒否できる）、B-02（誤認させる誘導＝dark pattern 禁止）。**ここはBaselineを超える倫理境界**。
- 観測指標: 継続率、完遂率、解約/解除の容易性（健全性チェック）。
- 注記: anticipatory design は理論先行で失敗例も多い。先回りが過剰だとエージェンシーを損ない逆効果。

## S-06 アクセシビリティ最大化

- 想定: 公共性が高い / 多様な利用者 / 法的要件のある画面。
- 狙い: WCAG AA超（AAA志向）と支援技術での完全操作。
- 強化: コントラストAAA、拡大耐性、スクリーンリーダー最適化、明確なfocus管理、平易な言葉。
- 緩めてよい: 視覚的リッチさ・装飾。
- 緩めてはならない: Baseline の知覚最低線（B-21 コントラスト〔§6〕＋ §7 アクセシビリティ floor B-22〜B-25）はそもそも最低線、ここで上限化。
- 観測指標: 支援技術での全タスク完了、自動/手動a11y監査の指摘ゼロ。

## 衝突解決

目的が衝突したとき、優先順位は固定する。

1. **安全境界（各Sの「緩めてはならない」）と §7 アクセシビリティ floor** — 常に最優先。いかなる目的でも侵食不可。
2. **誤りコストが高い操作の保護（S-02的要素）** — 速度・密度より優先。
3. **主目的の最適化レバー** — 上記に反しない範囲で。
4. **副目的** — 主目的と衝突したら主を採る。

例: ダッシュボード（S-04）に削除操作（S-02）が混在 → 密度は保ちつつ、削除のみ隔離・確認・undoを適用（2が4の密度都合に優先）。

## 運用メモ

- 1画面に目的を盛りすぎない。多目的画面は分割を検討する。
- 特化で緩めたBaselineは、緩めた理由とBアラインIDをコード/PRに残す（保守時の逆引き用）。
- 新しい特化目的を足すときは、必ず「強化／緩めてよい／緩めてはならない／観測指標」の4点を埋める。空欄のまま運用しない。
